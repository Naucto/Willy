# Willy ‚Äî Self-Hosted Mini-PaaS (master plan)

> Rework of `Naucto/Docker-Synchronizer` into **Willy**, a general self-hosted mini-PaaS.
> This file is the master plan. On implementation start it is copied to `docs/plans/0001-willy.md`
> and each phase below becomes (potentially) its own working session.

## Context

Today this repo is a single-purpose Python service: a Flask webhook listener (`hosting.py`) drives
`controller.py`, which pulls git submodules (`dev_environment` = the Naucto NestJS backend +
React/Vite frontend), runs `docker-compose build`, then `down`/`up`. It deploys **exactly one** app
to **one hardcoded domain** (`SSL_TARGET_DOMAIN`); TLS is **manual** (pre-existing certs mounted into
`fsouza/docker-ssl-proxy` sidecars); there is **no webhook authentication**; redeploys cause **full
downtime**; it is installed as a systemd service via `install.sh`.

We are **resetting** this repository into **Willy** (a nod to Docker's whale ‚Üí "Free Willy"): a
general mini-PaaS that deploys many arbitrary git apps ‚Äî each on its own domain/subdomain **or with
no domain at all** (background workers / cron jobs) ‚Äî with per-deployment encrypted env vars,
automatic Let's Encrypt certificates via OVH, OVH DNS record management, per-container console + log
journaling, volume/DB/S3 backup-restore, optional GitHub webhook auto-deploy, and a near-one-shot
remote provisioner that turns a fresh Debian VPS into a running Willy host. The existing Naucto app
stops being baked in and becomes *just one deployment Willy manages*; the GitHub‚ÜîJira synchronizer
and the Epitech repository-crawler mirror become **domain-less worker deployments**.

Why: the current system is a hardcoded, single-app, manual-TLS, full-downtime deployer. Naucto needs
to host multiple services (web apps *and* headless workers) across domains with self-service
management, automated certs, and operational tooling (logs, console, backups) from a web panel,
provisionable onto a fresh VPS in essentially one command.

**Network model (the key win):** one external network `willy_edge` that Traefik joins. WEB
containers join it and are routed **by Host rule, not host port** ‚Üí zero port collisions (the old
compose publishes `443`/`1987` to the host, only works for one app). WORKER/CRON containers do **not**
join `willy_edge` and get no labels. Compose stacks keep their internal project network; only the
declared web service attaches to `willy_edge`. Everything namespaced by `deploymentId` (+ git SHA
during swaps) to avoid Traefik router shadowing.

## Deployment type model (incl. domain-less)

`DeploymentType` enum drives everything downstream:
- **WEB** ‚Äî has ‚â•1 Domain; gets Traefik router/service labels on `willy_edge`; health probe via HTTP
  path; zero-downtime swap. (e.g. Naucto frontend/backend.)
- **WORKER** ‚Äî long-running, **no domain, no port, no Traefik labels, not on `willy_edge`**. Health =
  container running/healthcheck only. Swap = stop-old/start-new (brief gap acceptable; no traffic to
  drain). Full build/env/logs/console/backups/webhook support. (e.g. GitHub‚ÜîJira synchronizer, Epitech
  repository-crawler mirror.)
- **CRON** ‚Äî scheduled one-shot container on a cron expression (via `@nestjs/schedule`); no router;
  logs of each run retained. (Optional: a WORKER can declare an internal scheduler instead.)

The create-deployment wizard branches on type: WEB asks for port/domain/healthcheck; WORKER asks only
for run command/restart policy; CRON asks for the cron expression. Note: the two named apps (Jira sync,
repo crawler) still need their **own** code rewrites ‚Äî out of scope here; Willy only needs to *host*
them as WORKER deployments.

## Data model (Drizzle, `backend/src/db/`)

Schema defined as TypeScript via Drizzle `pgTable`/`pgEnum` in `backend/src/db/schema/*.ts` (one file
per aggregate), barrel-exported. Migrations generated + applied with **`drizzle-kit`** (`drizzle-kit
generate` ‚Üí versioned SQL in `backend/drizzle/`, `drizzle-kit migrate` to apply). IDs default to
`gen_random_uuid()` (pgcrypto) or cuid2; indexes/uniques expressed in the table builder.

**Enums** (`pgEnum`): `Role(ADMIN|OPERATOR|VIEWER)`, `DeploymentType(WEB|WORKER|CRON)`,
`BuildStrategy(NIXPACKS|DOCKERFILE|COMPOSE)`, `EnvScope(BUILD|RUNTIME|BOTH)`,
`ReleaseStatus(QUEUED|CLONING|BUILDING|HEALTHCHECKING|LIVE|SUPERSEDED|FAILED|ROLLEDBACK|INTERRUPTED)`,
`DeploymentState(CREATED|DEPLOYING|RUNNING|DEGRADED|STOPPED|ERROR)`,
`DomainType(SUBDOMAIN|CUSTOM_EXTERNAL|APEX)`, `CertStatus(NONE|PENDING|ISSUED|FAILED)`,
`DnsRecordType(A|AAAA|CNAME|TXT)`, `BackupKind(VOLUME_TAR|PG_DUMP|S3_SYNC)`,
`BackupStatus(PENDING|RUNNING|SUCCESS|FAILED)`, `RestartPolicy(NO|ON_FAILURE|ALWAYS|UNLESS_STOPPED)`,
`AuditAction(...)`.

**Models (key fields):**
- **User** ‚Äî email unique, `passwordHash` (argon2id), `role`, `refreshTokenHash`.
- **Deployment** ‚Äî `name` unique slug `^[a-z0-9][a-z0-9-]{0,40}$`, `type`, `gitUrl`, `gitRef`,
  `buildStrategy`, `dockerfilePath?`, `composeFilePath?`, `composeWebService?`, `runCommand?`,
  `cronExpr?`, `webServicePort?`, `healthCheckPath`, `autoDeploy`, `restartPolicy`, `memoryLimitMb?`,
  `nanoCpus?`, `state`, `activeReleaseId? @unique`. Relations: releases, envVars, domains,
  webhookSecret, backups.
- **Release** (== build job) ‚Äî `deploymentId`, `gitSha`, `imageTag` (`willy/<name>:<sha>`), `status`,
  `containerId?`, `composeProject?`, `logPath?`, `errorMessage?`, timestamps. `@@index([deploymentId, createdAt])`.
- **Domain** ‚Äî `fqdn @unique`, `deploymentId`, `type`, `certStatus`, `certResolver`, `isPrimary`.
- **DnsRecord** ‚Äî `zone`, `subDomain` (`""`=apex, `*`=wildcard), `type`, `target`, `ttl`,
  `ovhRecordId?`, `managedByWilly`, `deploymentId?`. `@@unique([zone, subDomain, type])`.
- **EnvVar** ‚Äî `deploymentId`, `key`, `cipherText`, `nonce`, `authTag`, `keyVersion`, `scope`,
  `isSecret`. `@@unique([deploymentId, key])`. Plaintext never stored/logged.
- **WebhookSecret** ‚Äî `deploymentId @unique`, encrypted secret triplet, `provider`, `lastDeliveryId?`.
- **GitCredential** ‚Äî auth for cloning **private app repos** at build time (distinct from Willy's own
  host-level deploy key set at provision time). `kind` (`PUBLIC|GITHUB_APP|DEPLOY_KEY|PAT`), encrypted
  payload (App installation id / private key, or per-repo SSH key, or token), optional `deploymentId`
  (or shared). **Recommended primary: a GitHub App** installed on the org ‚Üí short-lived per-clone
  installation tokens scoped to selected repos (no long-lived secrets, no per-repo key juggling);
  per-repo **deploy keys** and **PAT** as fallbacks. Stored encrypted via the same CryptoService as env vars.
- **Database** (managed DB instance) ‚Äî `name @unique`, `engine` (`POSTGRES|MYSQL|REDIS|MONGO`), `version`,
  `containerId?`, `volumeName`, `networkName` (private), encrypted credential triplet + `connStringEnvKey`,
  `state`, `memoryLimitMb?`. **DatabaseAttachment** ‚Äî `databaseId` √ó `deploymentId` (m:n), `injectPrefix`
  (env var prefix); attaching injects the encrypted connection string into the deployment's runtime env.
- **Backup** + **BackupSchedule** ‚Äî kind/status/target/location/sizeBytes/checksum/offsiteUrl;
  schedule: `cron`, `retention`, `enabled`, `lastRunAt`.
- **AuditLog** ‚Äî append-only; `actorId?`, `action`, `targetType/Id`, `metadata Json` (secret-scrubbed),
  `ip`, `createdAt`.

## Control plane (`backend/`) ‚Äî module map

Mirror existing Naucto NestJS conventions (Nest 11, `@nestjs/config` global, class-validator DTOs,
`@nestjs/swagger`, `@nestjs/schedule`, raw `ws` via `server.on("upgrade")` as in
`collab/signaling/signal.ts`) ‚Äî but swap Prisma for **Drizzle ORM** (a `DbModule` provides the typed
Drizzle client over a `pg` Pool; migrations via `drizzle-kit`). Upgrades over the old app: **argon2**
(not bcrypt) + **JWT+refresh**.

- **AuthModule** ‚Äî argon2, JWT access (~15m) + refresh (~7d, hashed on User for rotation), `JwtAuthGuard`,
  `RolesGuard` + `@Roles` (enum), `WsAuthService` for upgrade-time auth + stream tickets.
- **CryptoService** (in EnvVarsModule) ‚Äî AES-256-GCM, master key from `WILLY_MASTER_KEY` (validate on
  boot), per-secret 12-byte nonce, stored `{cipherText,nonce,authTag,keyVersion}`; **decrypt only at
  injection**; log scrubber.
- **DockerModule/DockerService** ‚Äî typed dockerode wrapper over `tcp://docker-socket-proxy:2375`:
  buildImage(+followProgress), createContainer/start/stop/remove, inspect (health + edge IP), logs
  (+`modem.demuxStream` for non-TTY), exec interactive TTY (+resize), network ensure/connect, scoped
  prune, image remove, ping, inspectVolume.
- **TraefikModule** ‚Äî `LabelGeneratorService` (literal `traefik.http.routers.<name>.*` /
  `services.<name>.*`, sha-namespaced during swap, multi-domain OR rule, **no host ports**;
  **skipped entirely for WORKER/CRON**) + `DynamicConfigService` (file-provider YAML for
  middlewares/dashboard/custom domains).
- **DeploymentsModule** ‚Äî CRUD + actions (deploy/redeploy/stop/start/rollback), `DeploymentStateMachine`,
  owns `activeReleaseId`. Branches behavior by `DeploymentType`.
- **BuildModule** ‚Äî `BuildQueue` iface + `InProcessBuildQueue` (semaphore + per-deployment mutex +
  coalescing of superseded queued jobs), `GitService` (shallow clone, SHA resolve, **SSRF URL
  validation**, **per-deployment `GitCredential` for private repos** ‚Äî GitHub App tokens / deploy key /
  PAT), `BuildOrchestrator` (state machine), strategies `NixpacksStrategy`/`DockerfileStrategy`/
  `ComposeStrategy`. Build-arg (`BUILD`/`BOTH`) vs runtime env (`RUNTIME`/`BOTH`) kept on separate paths.
- **DomainsModule** + **DnsModule** ‚Äî domain attach/detach + cert status; `OvhClient` (signed requests,
  `/auth/time`, record CRUD + `/refresh`, zone-scoped token).
- **EnvVarsModule** ‚Äî set/delete/listMasked/`resolveForInjection(scope)`; reveal endpoint (audited).
- **WebhooksModule** ‚Äî `POST /webhooks/github/:deploymentId`, **raw-body HMAC-SHA256** (constant-time),
  event=`push` + ref filter, `autoDeploy` gate, enqueue + coalesce.
- **ConsoleModule** ‚Äî WS exec TTY (hijacked duplex, resize), JWT at upgrade, `CONSOLE_OPEN` audit.
- **LogsModule** ‚Äî build-log SSE + runtime-log SSE/WS (demux non-TTY), tail + history + backfill.
- **DatabasesModule** ‚Äî managed DB lifecycle: create/start/stop/remove an engine container (Postgres/
  MySQL/Redis/Mongo) on a private network with a named volume + generated creds (encrypted); attach/detach
  to deployments (injects the connection string into runtime env + triggers a redeploy); exposes managed
  DBs as first-class backup targets (delegates to BackupsModule). Reconciled on boot like deployments.
- **BackupsModule** ‚Äî volume-tar / pg-dump / s3-sync via one-shot helper containers; restore lifecycle
  (stop writers ‚Üí optional pre-restore snapshot ‚Üí restore ‚Üí start ‚Üí healthcheck ‚Üí rollback-on-failure);
  `@nestjs/schedule` cron schedules; retention sweep; offsite push.
- **AuditModule**, **HealthModule** (`/health`, `/health/ready` = DB + docker ping),
  **ReconcileModule** (`OnApplicationBootstrap`: mark interrupted builds, recreate missing active
  containers, remove stale, re-sync Traefik file config ‚Äî **after** migrations).

### Build/deploy state machine
WEB health-checked swap: `QUEUED ‚Üí CLONING ‚Üí BUILDING (image willy/<name>:<sha>) ‚Üí` start new sha-named
container on `willy_edge` with labels+runtime env+limits `‚Üí HEALTHCHECKING` (container health + internal
HTTP probe at `healthCheckPath`; old still serving) `‚Üí` cutover (point Host rule at new) `‚Üí LIVE` (set
`activeReleaseId`) `‚Üí` stop+remove old (`SUPERSEDED`) `‚Üí` scoped cleanup (keep last N images). On
health-check failure: remove the new container, keep old live, mark `FAILED`. **Rollback** = start prior
`:<oldSha>` image + swap (no rebuild). **WORKER/CRON**: same up to BUILDING, then run with no labels/edge;
swap = stop-old/start-new. **Compose**: generate `willy.override.yml` (attach `composeWebService` to
external `willy_edge` + labels, **strip/reject host `ports:`**), `docker compose -p willy_<name> up -d
--build` (brief interruption), health-poll.

### Key backend gotchas
socket-proxy flags (`POST=1`,`EXEC=1`,`BUILD=1`,`CONTAINERS/IMAGES/NETWORKS/VOLUMES/INFO/PING=1`;
swarm/system/secrets off; never host-publish 2375) ¬∑ Traefik router name uniqueness ¬∑ build-arg vs
runtime env ¬∑ compose host-port stripping ¬∑ dockerode TTY (raw) vs non-TTY (demux) ¬∑ webhook raw body
before JSON parse ¬∑ `drizzle-kit migrate` as a one-shot entrypoint step (not racing in-process) ¬∑ never global
prune (keep-N + dangling-only) ¬∑ **acme.json + master key are catastrophic to lose** (back up both).

## Traefik / routing (`routing/`) ‚Äî `traefik:v3.7`

**Per-domain certs across any OVH zone is the primary model** (apps aren't confined to the panel zone).
Static `routing/traefik.yml`: entrypoints `web:80`‚Üíredirect‚Üí`websecure:443`; providers docker
(`exposedByDefault=false`, endpoint `tcp://docker-socket-proxy:2375`, network `willy_edge`, watch) +
file (`/etc/traefik/dynamic`); `certificatesResolvers.ovh.acme` DNS-01 (OVH provider, resolvers
`1.1.1.1`/`8.8.8.8`, `propagation.delayBeforeChecks 30s`, **staging caServer first**); `api.dashboard
true insecure:false`. OVH creds + `OVH_PROPAGATION_TIMEOUT` as Traefik env. Each WEB router carries
`tls.certresolver=ovh`, so Traefik issues a cert **per FQDN on demand** via OVH DNS-01 ‚Äî works for any
zone the OVH token controls.

**Optional per-zone wildcard optimization:** for a zone with many subdomains (e.g. the panel zone
`willy.naucto.net`), pre-declare a `tls.domains` entry (main `willy.naucto.net` + san `*.willy.naucto.net`)
so those subdomains resolve to one cert and skip per-host challenges. Willy maintains these wildcard
entries via the file provider as zones accumulate apps; everything else falls back to per-domain certs.

Per-WEB-container labels (literal): `traefik.enable=true`, `traefik.docker.network=willy_edge`,
`routers.<name>.rule=Host(\`<app-fqdn>\`)`, `.entrypoints=websecure`, `.tls=true`, `.tls.certresolver=ovh`,
`.service=<name>`, `.middlewares=sec-headers@file`, `services.<name>.loadbalancer.server.port=<PORT>`.
File-provider dynamic: `middlewares.yml` (sec-headers HSTS, panel-auth basicAuth, rate-limit),
`dashboard.yml` (`traefik.<panel-zone>` ‚Üí `api@internal` behind panel-auth), `wildcards.yml` (managed
per-zone wildcard `tls.domains`). Panel routers on willy-web/willy-server (`willy.naucto.net` + API under
`/api` or `api.` subdomain); SSE/WS pass through (no buffering middleware on those routers).

**Domain validation ("OVH perimeter"):** at domain-add, Willy resolves the FQDN's registrable zone and
checks it exists in the OVH account / token scope; if not, reject (or warn) since DNS records + DNS-01
can't be automated outside it.

Gotchas: router/service name uniqueness ¬∑ a wildcard covers **one** label only (no `a.b.zone`; apex via
`main`) ‚Üí deeper depths need their own cert ¬∑ staging‚Üíprod = change caServer **and delete acme.json**
then restart ¬∑ LE limits (50 certs/registered-domain/wk ‚Üí prefer per-zone wildcards for high-subdomain
zones; per-domain certs for the long tail; **always validate on staging**) ¬∑ OVH propagation slow
(generous timeout) ¬∑ acme.json **chmod 600** or Traefik aborts.

## Backups (`backend/` BackupsModule) ‚Äî containerize the team's existing approach

Reuses `~/Scripts/nc_db_backup.sh` (volume mountpoint tar) and `nc_s3_backup.sh` (aws s3 sync) but runs
**on the host via the control plane** (no SSH) by mounting volumes into one-shot helper containers:
- **volume-tar**: `docker run --rm -v <vol>:/data:ro -v willy_backups:/backup alpine tar -czf
  /backup/<id>.tar.gz -C /data .` (replaces `docker volume inspect {{.Mountpoint}}` + `cp -a`). Restore:
  stop writers ‚Üí wipe+extract ‚Üí start writers.
- **pg-dump**: `postgres:<major>` helper `pg_dump -Fc` / `pg_restore --clean --if-exists` (version-safe,
  unlike raw data-dir tar).
- **s3-sync**: `amazon/aws-cli s3 sync` (+`--endpoint-url` for non-AWS, reusing repo `S3_ENDPOINT`).
Storage on `willy_backups` volume (`/<deploymentId>/<kind>/<id>`), optional offsite S3 push, sha256 +
size + `Backup` row; `BackupSchedule` cron via `SchedulerRegistry`; retention keep-N sweep; single-flight
per target; typed-name confirm + server-side `confirm` flag; everything audited.

## Remote provisioner (`scripts/provision.sh` + `scripts/willy_deploy.sh`)

Two POSIX-sh files (port the `install.sh` idioms `sv_require`/`sv_try`/`sv_question`/`sv_lock`,
`/dev/urandom`/`openssl rand` secrets, verb dispatch). `provision.sh` runs on the operator laptop,
preflights, `scp`s `willy_deploy.sh` + a generated `.env`, runs it via `ssh ‚Ä¶ 'sudo sh -s'`. Verbs:
`provision` (default), `upgrade`, `doctor`, `backup`. (Ansible/cloud-init/Go-CLI evaluated and rejected
for a one-host op; revisit Ansible past ~3 hosts.)

- **Inputs** (flags + env-only for secrets): `--host/--ssh-user/--ssh-key/--ssh-port`, `--base-domain`,
  `--panel-subdomain` (default `panel`), `--acme-email`, `--ovh-endpoint` + `OVH_APPLICATION_KEY/SECRET`
  + `OVH_CONSUMER_KEY` (env only), `--admin-email` + `WILLY_ADMIN_PASSWORD?` (gen+print-once if absent),
  `--repo/--ref`, `--ca staging|prod` (default staging‚Üíauto-flip).
- **Preflight**: local tools; OVH creds present; SSH reachable + non-interactive sudo; Debian ‚â•12
  (Ubuntu path branch); **DNS resolves panel/apex/wildcard-canary ‚Üí host IP**; ports 80/443 free; disk
  ‚â•5GB; clock skew.
- **Host setup (fresh Debian)**: `DEBIAN_FRONTEND=noninteractive`; apt update/upgrade; base pkgs
  (`ca-certificates curl gnupg git ufw fail2ban unattended-upgrades jq openssl`); **Docker official
  repo** (keyring + `$VERSION_CODENAME`, debian vs ubuntu path) + `docker-ce docker-ce-cli containerd.io
  docker-buildx-plugin docker-compose-plugin`; enable docker; `/etc/docker/daemon.json` log rotation
  (first run only); `willy` user in `docker` group (group-refresh gotcha ‚Üí run docker steps as root /
  fresh session); ufw deny-default + allow 22/80/443; unattended security upgrades (no surprise reboot);
  journald size cap; swap if RAM<2GB; SSH hardening drop-in (password off) **after** key-auth verified.
- **Secrets + .env**: idempotent `set_secret KEY GEN` (preserve existing, never clobber `WILLY_MASTER_KEY`/
  DB pw/JWT on re-run); generate via `openssl rand`; atomic mode-600 willy-owned `.env`; never echo
  secrets; admin password hashed by the app's own seed (algorithm parity), printed once only if generated.
- **Deploy**: clone/reset `/opt/willy@ref`; ensure `letsencrypt/acme.json` mode 600; `docker compose
  pull` (+`build`); `up -d`; wait healthchecks (postgres/willy-server/traefik); `drizzle-kit migrate`;
  **idempotent admin upsert seed**; cert poll on **staging** then **flip to prod** (rewrite caServer,
  empty acme.json, restart traefik, verify issuer ‚âÝ STAGING via `openssl s_client`).
- **upgrade**: lite preflight ‚Üí git reset ‚Üí `compose pull` ‚Üí `up -d --remove-orphans` (only changed
  services bounce; tenant apps untouched) ‚Üí `drizzle-kit migrate` ‚Üí health + doctor. Remote lockfile.
- **doctor** (read-only, non-zero on any ‚úó): DNS, OVH API reachability, container health, cert
  perms/issuer/expiry, disk + `docker system df`, ports, user/group, time skew, `.env` sanity.
- **Security**: SSH key-only; OVH token **zone-scoped** (`GET/POST/PUT/DELETE /domain/zone/<zone>/*` +
  `/refresh`); secrets in `/opt/willy/.env` (600) + acme.json (600); `backup` verb bundles pg_dump + `.env`
  + acme.json off-host (encrypt); ufw; docker-socket-proxy is sole socket mount.

### Operator runbook (the "almost one shot")
Concrete Naucto target: **panel** at `willy.naucto.net`. Apps deploy to **any** FQDN in an OVH-managed
zone (e.g. `app.willy.naucto.net`, `crawler.naucto.net`, `staging.someproduct.io` if that zone is in
OVH) ‚Äî chosen per app in the UI, no reprovision needed.
1. Provision a fresh Debian 12 VPS; install your SSH key for root (or passwordless-sudo user); note IP.
2. Create an OVH API token scoped to **every zone you'll deploy into** (at minimum `naucto.net`):
   `GET/POST/PUT/DELETE /domain/zone/<zone>/*` + `/refresh`. Save app key/secret + consumer key.
3. Point the **panel** record: `willy.naucto.net A <IP>` (and, if you want a fast wildcard for
   subdomains of that zone, `*.willy.naucto.net A <IP>`); wait until `dig +short willy.naucto.net` == `<IP>`.
   Per-app A/CNAME records are created later **from the panel** when you add a domain to a deployment.
4. One command: `OVH_APPLICATION_KEY=‚Ä¶ OVH_APPLICATION_SECRET=‚Ä¶ OVH_CONSUMER_KEY=‚Ä¶ ./scripts/provision.sh
   --host <IP> --ssh-user root --base-domain willy.naucto.net --panel-subdomain "" --acme-email you@‚Ä¶
   --ovh-endpoint ovh-eu --admin-email you@‚Ä¶`
5. Log in at `https://willy.naucto.net` with the printed one-time admin password; change it. Then deploy
   apps to whatever OVH-managed domains you like.

(`--base-domain` only sets the **panel**'s home + the default zone offered first in the new-deployment
wizard; apps are free to use other OVH zones.)
‚Üí 3 manual setup steps + 1 command; only DNS propagation + human OVH-token creation prevent literal one-shot.

## Version switching (health-checked swap, brief interruption)

**Not strict zero-downtime ‚Äî a brief interruption is acceptable** (per user). All triggers (manual deploy,
webhook auto-deploy, rollback) use one path: build/resolve the new immutable image ‚Üí start the new
container ‚Üí **health-check it** (container health + internal HTTP probe at `healthCheckPath`) ‚Üí flip the
Traefik Host router to the new container ‚Üí stop + remove the old. The only safety property kept: the **old
version keeps running until the new one passes health-check**, so a failed build/boot never takes down the
working version (on failure: discard the new container, keep the old, mark the deploy failed). The sole
"downtime" is the sub-second route flip ‚Äî no side-by-side dual-serving, no connection draining, no
zero-failed-requests guarantee. WORKER/CRON have no inbound traffic (stop-old/start-new). Compose stacks
recreate in place.

## Local development & testing

Willy runs end-to-end on a laptop with **no VPS, no OVH, no public DNS** ‚Äî the default way to develop
and to "test it out".

- **Domains:** base domain `localhost` ‚Üí panel at `willy.localhost`, deployed apps at `<app>.localhost`.
  Browsers resolve `*.localhost` to `127.0.0.1` automatically (no `/etc/hosts` edits needed).
- **Certs:** `docker-compose.local.yml` swaps the OVH/ACME cert resolver for a **mkcert**-generated
  wildcard cert for `*.localhost`, mounted into Traefik via the file provider ‚Üí browser-trusted HTTPS
  with no warnings. (Fallback: Traefik default self-signed if mkcert isn't installed.) The OVH/DNS-01
  code path is simply not exercised locally.
- **Parity:** same Traefik + docker-socket-proxy + Postgres + willy-server/web containers and the same
  build/deploy/swap/console/logs/backup logic, run against the **local Docker daemon**. Deploying apps
  through Willy locally behaves exactly like prod (minus public reachability).
- **Run:** `make dev` = `docker compose -f docker-compose.yml -f docker-compose.local.yml up`. Two flavors
  via the override: **parity** (built images) and **watch** (Vite dev server + `nest start --watch`,
  source-mounted, for fast iteration) ‚Äî pick with a compose profile.
- **Seed:** local bootstrap auto-creates an admin (`admin@willy.localhost` / known dev password) and the
  mkcert cert, so `make dev` ‚Üí log in immediately.
- **Advanced (optional):** to test the *real* OVH cert path locally, DNS-01 issues certs without inbound
  reachability, so you can point a real OVH FQDN at `127.0.0.1` via `/etc/hosts` and get a genuine LE
  **staging** cert ‚Äî documented but not the default.
- **Tests run locally too:** `make test` (Vitest), `make e2e` (Playwright drives `willy.localhost`),
  integration via Testcontainers ‚Äî all without a VPS.

This is delivered in **Phase 1** so every later phase is built and tested locally first; the VPS is for
integration/regression (Phase 8) and real deployments.

## Out of scope (v1)

- **Multi-host / clustering** ‚Äî single-host only. Likely not needed now or even later; revisit only if
  real demand appears (would then need an image registry + scheduler + shared state).
- **Non-OVH DNS providers** ‚Äî OVH is the only DNS/ACME integration; other providers are out of scope.
- **Code rewrites of other Naucto services** (GitHub‚ÜîJira synchronizer, Epitech repo-crawler) happen in
  **their own repos** ‚Äî explicitly *not* part of Willy. Willy only needs to *host* them as WORKER
  deployments; nothing about those rewrites lands in this repo.

**Switching model (by choice, not a gap):** all version switches accept a **brief interruption** (the
sub-second Traefik route flip; compose stacks a bit more). Strict zero-downtime (side-by-side overlap +
connection draining) is intentionally **not** implemented. The kept safety property is that a broken
build never replaces a healthy running version (old stays up until new passes health-check).
